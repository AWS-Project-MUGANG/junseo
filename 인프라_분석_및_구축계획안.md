# AI 학사행정(그룹웨어) 인프라 분석 및 구축 계획서

본 문서는 강사님의 전체적인 요구사항(요건 만들기 -> 증적)을 프로젝트에 부합하도록 구성한 아키텍처 및 인프라 구축 분석서입니다.

## 1. 요건 만들기 (Requirements Definition)
* **목표**: AI IaaS 리소스를 활용한 고도화된 학사행정 및 그룹웨어 서비스 구축.
* **주요 서비스**: 
  1. Frontend: Node.js (React/Next.js 기반 혹은 바닐라 JS 배포)를 이용한 웹 서비스.
  2. Backend: Python(FastAPI) 및 Java 연동 MSA 기반 API 서비스.
  3. AI 서비스: RAG(Pinecone, AWS Bedrock 등) 기반 문서 검색 및 AI 챗봇 서비스.
* **인프라 요구사항**: 
  * Docker를 활용한 컨테이너화 및 Kubernetes(EKS)를 이용한 MSA 관리 및 오케스트레이션.
  * 하이브리드 클라우드와 다양한 AWS 리소스(VPC, API Gateway, S3, RDS 등) 통합.
  * IaC 도구(Ansible, Terraform, CloudFormation, AWS CLI, Shell)를 이용한 인프라 자동화 및 구성 관리 도입.

## 2. 요건 분석 (Requirements Analysis)
* **애플리케이션 계층**:
  * AI 모듈(Python)과 안정성 위주의 비즈니스 로직(Java/Python)을 분리하여 MSA(마이크로서비스 아키텍처) 구조 채택 필요. (서로 다른 부하 특성 대응)
* **네트워크/보안 (VPC & API GW)**:
  * 인바운드 트래픽은 AWS API Gateway 단일 진입점으로 통제하며, EKS 클러스터는 프라이빗 서브넷에 격리하여 보안 강화.
* **DevOps 프로세스**:
  * 소스 관리(GitHub)부터 빌드(Docker), 저장(ECR), 배포(EKS/ArgoCD)까지를 아우르는 오픈소스 및 클라우드 매니지드 서비스 조합 구성.

## 3. 설계 [인프라 및 아키텍처] (Infrastructure Design)
TA(Technical Architect) 관점에서 구성한 인프라 청사진입니다.

* **네트워크**: 1개의 VPC 내에 Public Subnet (ALB/NAT GW 위치)과 Private Subnet (EKS Worker Node, RDS, ElastiCache 위치) 분리 구성.
* **진입점 및 라우팅**: Route 53 -> WAF -> API Gateway -> ALB (Ingress Controller).
* **마이크로서비스 (EKS 기반)**:
  * 프론트엔드 Pod (Nginx/Node)
  * 백엔드 API Pod (FastAPI, Java Spring Boot)
* **데이터 베이스**: Amazon RDS (PostgreSQL) - Multi-AZ 적용 고려.
* **AI & 데이터 파이프라인**: Amazon S3(문서 원본) -> AWS Lambda(청킹/임베딩) -> Pinecone / Amazon OpenSearch(벡터 검색) -> Bedrock(LLM).
* **IaC 도구 활용안**:
  * **Terraform / CloudFormation**: VPC, EKS 클러스터, RDS 등 베이스 인프라 프로비저닝.
  * **Ansible**: 초기 리눅스 노드 셋업, 하이브리드 구성 시 온프레미스 서버 환경 일괄 구성 (Docker 데몬, 에이전트 설치 등).
  * **AWS CLI & Shell 스크립트**: CI/CD 파이프라인 내부의 ECR 로그인, Docker build & push, 배포 롤백 등의 자동화.

## 4. 구축 (Implementation Strategy)
1. **1단계 (IaC 프로비저닝)**: Terraform으로 인프라 코드를 작성해 AWS 환경(네트워크, EKS, RDS) 일괄 생성.
2. **2단계 (CI/CD 구성)**: GitHub Actions 로 파이프라인을 구축하여 코드 푸시 시 Docker Image 빌드 후 ECR 저장.
3. **3단계 (EKS 배포)**: Kubernetes Manifest(Yaml) 파일 혹은 Helm(오픈소스)을 이용해 EKS 클러스터에 배포. Blue-Green 배포 전략 사용.
4. **4단계 (AI 파이프라인)**: S3와 Lambda 트리거를 연동하고 API Gateway와 백엔드 API를 통합.

## 5. 테스트 및 증적 가이드 (Testing & Evidence)
개발 및 구축이 끝난 후, 다음 항목들을 캡처하여 **Markdown(md) 형식**으로 증적을 남깁니다.
1. **인프라 증적**: Terraform Apply/CloudFormation 생성 완료 터미널 화면 및 AWS 콘솔 캡처(VPC, EKS, ECR).
2. **배포 증적**: GitHub Actions 배포 성공(초록색 체크) 화면, `kubectl get pods` 상태가 Running인 화면.
3. **기능 증적**: API Gateway를 통한 회원가입 API 테스트(Postman) 화면 및 AI 챗봇(웹) 응답 화면 캡처.
4. **보안 증적**: Security Group의 Inbound 규칙 콘솔 화면 캡처.

## 6. 리소스, 일정(WBS), 비용, 용량 산정
### 6.1 용량 및 리소스 산정 (초기 구축 기준)
* **EKS Node Group**: `t3.medium` 또는 `m5.large` (2식 - 가용영역 분산) - AI 모듈 및 웹 서비스 구동 감당.
* **RDS (DB)**: `db.t3.micro` (개발용) / `db.t3.medium` (운영용) + 20GB SSD.
* **Storage**: Amazon S3 Standard (문서 저장용 10GB 시작), ECR 컨테이너 레지스트리 (10GB).
* **Network**: 로드밸런서(ALB) 1개, NAT Gateway 1개.

### 6.2 예상 비용 (월별 러프 산정)
* Amazon EKS Control Plane: 약 $73
* EC2 (t3.medium 2대): 약 $60
* RDS (t3.micro): 약 $15 ~ $30
* NAT Gateway & ALB: 약 $50
* S3, Data Transfer, Bedrock API 호출 비용: 종량제 (약 $20)
* **총합**: 최소 월 약 $220 ~ $300 수준 (개발 환경 기준). 실제 운영용은 스케일링에 따라 변동.
