# 무강대학교 AI 학사행정 배포용 AWS 스터디 가이드 (기업 요구 명세서 반영)

이 문서는 본 프로젝트를 AWS 클라우드 환경에 MSA, EKS(블루-그린) 방식으로 배포하기 위해 강사님(또는 프로젝트 발표자)이 직접 수행해야 할 AWS 인프라 세팅 절차와 아키텍처 설명서입니다.

---

## 1. 달성 목표 아키텍처 요약
이번 프로젝트는 단순 EC2 서버 배포가 아닌 **[MSA 기반 컨테이너 무중단 배포]**를 목표로 합니다.
1. **Frontend (바닐라 JS + Nginx)**: 사용자의 수강신청/시간표 화면 담당.
2. **Backend (Python FastAPI)**: 회원 정보 검증, 수강 장바구니 DB 저장, AI RAG 문서 검색 API 담당.
3. **CI/CD 파이프라인 (GitHub Actions ➡️ ECR ➡️ EKS)**: 개발자가 깃허브에 코드를 푸시하면, 깃허브가 자동으로 도커 이미지를 구워서 ECR(저장소)에 업로드하고 EKS(쿠버네티스) 서버들에 새로운 코드를 무중단(블루-그린)으로 덮어씌웁니다.

---

## 2. 배포 전 [사전 클릭 작업] 목록 (AWS 콘솔)
제가 짠 `.github/workflows/deploy.yml` 자동화 로봇 서버가 작동하려면, 강사님이 먼저 AWS 홈페이지에서 아래 자원과 열쇠를 만들어 주셔야 합니다.

### Step 1: 저장소(Amazon ECR) 2개 만들기
* **할 일**: 코드를 감싼 도커(Docker) 상자를 보관할 창고 만들기.
* **방법**: AWS ECR(Elastic Container Registry) 접속 👉 [리포지토리 생성] 클릭 👉 `mugang-frontend`와 `mugang-backend`라는 이름으로 컨테이너 리포지토리 2개 생성.
* **결과**: `123456789.dkr.ecr.ap-northeast-2.amazonaws.com/mugang-frontend` 처럼 생긴 긴 주소 2개 확보.

### Step 2: 배포 무대(Amazon EKS 클러스터) 만들기
* **할 일**: 실제로 프로그램이 구동될 쿠버네티스(Kubernetes) 서버 군단 생성.
* **방법**: AWS EKS 접속 👉 [클러스터 생성] 클릭 👉 이름(예: `mugang-eks-cluster`) 지정 후 기본 옵션으로 생성 (생성에 약 10~15분 소요).

### Step 3: 출입 암호 (IAM 사용자 Access Key) 발급받기
* **할 일**: 깃허브 로봇 서버가 강사님의 AWS ECR/EKS를 원격으로 조종할 수 있게 출입증 만들기.
* **방법**: AWS IAM 접속 👉 사용자 추가 (예: `github-actions-user`) 👉 권한 추가(ECR PowerUser, EKS Admin 등) 👉 [보안 자격 증명] 탭에서 **액세스 키(Access Key ID)**와 **비밀 액세스 키(Secret Access Key)** 발급.
* **중요**: 이 두 가지 키 문자열은 깃허브 레포지토리의 `Settings > Secrets and variables > Actions` 메뉴에서 `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` 항목으로 저장해 두어야 자동 배포 코드가 작동합니다.

## 3. 포트 제어(방화벽) 및 보안 인증 (Security Group)
명세서 요구사항의 "보안 인증은 시큐리티 그룹이나 스케줄된 포트 제어로 방화벽 역할 세팅" 지침에 따른 세팅 방향.

1. **프론트엔드 EKS 로드밸런서 (ALB)**: 
   - 접속 허용(Inbound): 전 세계 어디서든(0.0.0.0/0) **HTTP(80) 및 HTTPS(443)** 포트만 허용.
2. **백엔드 (FastAPI) 컨테이너**: 
   - 접속 허용: 프론트엔드와 AWS 내부 API Gateway에서 들어오는 요청(포트 8000)만 허용하도록 시큐리티 그룹을 빡빡하게 설정하여 외부에서의 다이렉트 해킹 방지.
3. **Database (RDS - PostgreSQL)**:
   - 접속 허용: 백엔드 컨테이너가 속해있는 시큐리티 그룹만 DB(포트 5432)에 접근할 수 있도록 보안 처리.

## 4. 포트폴리오(자소서) 스토리텔링 팁
> "이 기술을 왜 사용했는지에 대한 점을 자기소개서에 스토리텔링 필요"

개발 직무 또는 클라우드 엔지니어로 면접에 가실 때에는 다음 포인트들을 강조하시면 좋습니다.

* **왜 리액트를 버리고 바닐라 JS(Vanilla JS)를 썼나요?**
  "프론트엔드 프레임워크의 무거운 의존성 설치 문제를 배제하고, 가장 빠르고 근본적인 웹 표준 기술만으로 가벼운 Nginx 도커 환경을 구축하기 위해서입니다."
* **왜 EC2가 아니라 EKS(Kubernetes)와 MSA 환경을 썼나요?**
  "수강신청 시즌이나 학사 공지글이 터질 때 사용자 트래픽이 폭주하는데, 쿠버네티스가 트래픽 부하를 감지해 자동으로 서버 대수를 늘려주는 스케일아웃(Auto Scaling)에 유리하고, 멈춤 없이 신버전을 패치(블루-그린 배포)하기 위함입니다."
* **왜 AI 모듈 배포 시점과 프론트 코어를 분리(MSA)했나요?**
  "RAG 기반 (Pinecone/Bedrock) 질문답변 모듈은 연산(Token) 부하가 심한 반면, 일반 '수강신청/장바구니' 모듈은 빠르고 안정적인 DB 처리가 생명입니다. 두 시스템이 하나의 서버에 묶여있으면 AI 서칭 작업 때문에 수강신청이 먹통이 될 수 있어서 MSA 파드(Pod) 구조로 분리했습니다."

## 5. 개발 환경 트러블슈팅

### uvicorn 서버가 Ctrl+C로 안 꺼질 때

터미널에서 `Ctrl+C`가 먹히지 않는 경우, PowerShell로 포트를 점유한 프로세스를 강제 종료한다.

```powershell
# 1단계: 포트 8000을 쓰는 PID 확인
netstat -ano | findstr :8000

# 2단계: 해당 PID 강제 종료 (예: PID가 1152인 경우)
powershell -Command "Stop-Process -Id 1152 -Force"
```

> **왜 Ctrl+C가 안 되나?** Windows에서 uvicorn을 `--reload` 옵션으로 실행하면 부모/자식 프로세스가 분리되어, 터미널 시그널(SIGINT)이 자식 프로세스에 전달되지 않는 경우가 있다.
